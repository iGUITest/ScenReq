# ScenReq 需求交付与专家修改规范

> 版本：v2.0  
> 日期：2026-05-12  

---

## 一、交付物

系统为每个 APP 生成一份 **Excel 工作簿**（`{app}_需求整合.xlsx`），包含 4 个 Sheet：

| Sheet | 内容 | 专家操作 |
|-------|------|---------|
| 功能需求(SRS) | 系统生成的全部功能需求 | 参考用，一般不改 |
| 功能需求(过滤后) | 筛选出适合测试的功能需求 | 参考用 |
| 测试需求 | 按功能需求生成的测试用例 | 参考用 |
| **功能需求-测试需求对应** | **功能需求与测试用例的整合视图** | **主要修改对象** |

> 直接将 Excel 文件交给专家修改，无需接触 JSON。

---

## 二、"功能需求-测试需求对应" Sheet 字段说明

| 列名 | 含义 | 可修改 |
|------|------|--------|
| 功能需求编号 | 功能需求唯一 ID（如 FR-001） | ❌ 不改 |
| 功能需求名称 | 功能需求标题 | ❌ 不改 |
| 功能需求描述 | 功能需求完整描述 | ❌ 不改 |
| 功能需求类型 | 需求分类（功能性/界面/性能等） | ❌ 不改 |
| 测试编号 | 测试用例 ID（如 TC-01-01） | ❌ 不改 |
| 测试用例名称 | 测试场景标题 | ✅ 可修改 |
| 测试描述 | 操作步骤与场景描述 | ✅ **核心修改字段** |
| 测试目标 | 验证目的（以"验证..."开头） | ✅ 可修改 |
| 前置条件/数据 | 测试所需前置条件或输入数据 | ✅ 可修改 |
| 预期结果 | 判定通过/失败的可观测标准 | ✅ **核心修改字段** |
| 操作步骤 | 详细操作步骤 | 可选进行描述 |

---

## 三、修改原则

1. **只改测试相关列**：不动功能需求编号、名称、描述、类型这 4 列
2. **测试描述要具体**：包含操作者 + 操作动作 + 操作对象 + 上下文
   - ❌ "测试登录功能" → ✅ "用户在登录页输入已注册手机号和正确密码，点击登录按钮"
3. **预期结果要可观测**：避免"功能正常"等模糊表述
   - ❌ "登录成功" → ✅ "跳转至主会话页，顶部显示用户昵称和头像"

## 四、Skill 编写规范

> Skill 由专家直接编写，是一份 **Markdown 文档**（`.md`），用于指导 GUI Agent 执行测试。

### 格式

```markdown
---
name: login-benchmark
version: 1.0.0
description: 登录功能全场景 Benchmark
tags:
  - benchmark
  - login
---

# 登录功能 Benchmark

#### L-001 手机号+密码登录 — 正确凭据

- **前置条件**: 用户已注册账号，手机号 138xxxx1234，密码 P@ssw0rd123
- **操作步骤**:
  1. 进入登录页面
  2. 输入正确手机号
  3. 输入正确密码
  4. 点击「登录」按钮
- **预期结果**: 登录成功，跳转至首页
```

### 结构说明

- **YAML 头部**：`name`（标识名）/ `version` / `description` / `tags`
- **场景格式**：`#### {场景ID} {场景名称}`，内含前置条件、操作步骤、预期结果
- **场景 ID**：建议格式 `{前缀}-{编号}`，如 `L-001`、`S-028`
- **存储位置**：`skills/{app_name}/SKILL.md`
