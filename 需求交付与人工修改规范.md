# ScenReq 需求交付与人工修改规范

> 版本：v1.0  
> 日期：2026-05-07  
> 说明：本文档定义系统生成需求的交付物格式、专家修改范围、以及修改后的保存回写方式。供交接与协作使用。

---

## 一、系统输出交付物清单

每处理一个 APP，系统在对应输出目录下生成以下文件：

| 文件名 | 格式 | 内容 | 人工可修改 |
|--------|------|------|-----------|
| `{app}_SRS.json` | JSON | 软件需求规格（功能需求列表） | ❌ 直接改 JSON 不推荐，改 Word 后回写 |
| `{app}_TestRequirements.json` | JSON | 测试需求列表（测试用例级） | ❌ 同上 |
| `{app}_功能测试报告.docx` | Word | 综合报告，含 SRS 表 + 测试需求表 | ✅ 专家在此文件中修改 |
| `{app}.json`（test_json） | JSON | 测试场景文件（供自动化执行） | ⚠️ 一般由系统生成，特殊情况下可手动调整 |

> 文件命名关联：`xxx_功能测试报告.docx` ↔ `xxx_SRS.json` ↔ `xxx_TestRequirements.json`，三文件位于同一目录。

---

## 二、JSON 数据结构（系统原始输出）

### 2.1 软件需求：`{app}_SRS.json`

数组结构，每条需求为一个对象：

```json
[
  {
    "id": "FR-001",
    "name": "主界面展示欢迎信息与功能入口",
    "requirement": "系统应在应用启动后立即在主界面显示清晰的欢迎文案...",
    "type": "功能性"
  },
  {
    "id": "FR-002",
    "name": "网页内链跳转功能",
    "requirement": "系统应在网页浏览过程中支持用户点击页面内任意超链接...",
    "type": "功能性"
  }
]
```

| 字段 | 含义 | 是否必填 | 专家可修改 |
|------|------|----------|-----------|
| `id` | 需求唯一编号 | 是 | ⚠️ 不建议改，作为合并键 |
| `name` | 需求名称/标题 | 是 | ✅ 可修改 |
| `requirement` | 需求详细描述 | 是 | ✅ 可修改 |
| `type` | 需求类型 | 是 | ✅ 可修改（功能性/界面/性能/安全等） |

### 2.2 测试需求：`{app}_TestRequirements.json`

数组结构，每条测试需求为一个对象：

```json
[
  {
    "id": "TC-01-01",
    "name": "正常输入数值进行米转厘米转换",
    "description": "用户在长度转换页面选择米为输入单位，厘米为输出单位...",
    "objective": "验证系统能正确完成米到厘米的单位换算",
    "extra_info": "输入值：5",
    "expected_result": "显示正确的换算结果（1米 = 100厘米）"
  }
]
```

| 字段 | 含义 | 是否必填 | 专家可修改 | 备注 |
|------|------|----------|-----------|------|
| `id` | 测试用例编号 | 是 | ⚠️ 不建议改 | 合并键，格式 `TC-模块-序号` |
| `name` | 用例名称 | 是 | ✅ | 简明描述测试点 |
| `description` | 测试步骤/场景描述 | 是 | ✅ | 详细操作步骤 |
| `objective` | 测试目标 | 是 | ✅ | 验证目的 |
| `extra_info` | 额外信息/测试数据 | 否 | ❌ | **Word 不展示，回写时保留原值** |
| `test_data` | 测试输入数据 | 否 | ✅ | 具体输入值 |
| `expected_result` | 预期结果 | 是 | ✅ | 明确的通过标准 |

> **重要**：`extra_info` 字段在 Word 报告中不显示，专家修改 Word 时无法触及。回写时会自动保留原始 JSON 中的 `extra_info`，不会被覆盖或清空。

### 2.3 测试场景：`test_json/{app}.json`

```json
{
  "apps": [
    {
      "id": "A1",
      "name": "com.ferrarid.converterpro",
      "package": "com.ferrarid.converterpro",
      "launch-activity": "com.ferrarid.converterpro.MainActivity"
    }
  ],
  "scenarios": [
    {
      "id": "S_主界面正常加载欢迎文案与功能图标_001",
      "name": "主界面正常加载欢迎文案与功能图标",
      "description": "应用启动后，主界面完整显示欢迎文案及三个常用转换类别图标...",
      "extra-info": { "value": "无" }
    }
  ]
}
```

| 字段 | 含义 | 专家可修改 |
|------|------|-----------|
| `apps` | 被测应用信息 | ⚠️ 一般不动 |
| `scenarios` | 测试场景列表 | ⚠️ 一般由系统生成，特殊场景可手动增删 |

> 测试场景文件通常由系统直接从 SRS/测试需求转换生成，**不属于 Word 回写范围**。如需修改，直接编辑 JSON。

## 三、Word 报告结构（专家修改的入口）

### 3.1 文件内容

Word 报告包含两个表格：

**表 1：软件需求规格（SRS）**

| ID | Name | Requirement | Type |
|----|------|-------------|------|
| FR-001 | 主界面展示欢迎信息与功能入口 | 系统应在应用启动后立即在主界面显示清晰的欢迎文案... | 功能性 |
| FR-002 | 网页内链跳转功能 | 系统应在网页浏览过程中支持用户点击页面内任意超链接... | 功能性 |

**表 2：测试需求**

| ID | Name | Description / Objective | Test Data | Expected Result |
|----|------|------------------------|-----------|-----------------|
| TC-01-01 | 正常输入数值进行米转厘米转换 | 用户在长度转换页面选择米为输入单位... / 验证系统能正确完成米到厘米的单位换算 | 输入值：5 | 显示正确的换算结果 |

### 3.2 Word ↔ JSON 字段映射

```
表 1（SRS）
  Word: ID        → JSON: id
  Word: Name      → JSON: name
  Word: Requirement → JSON: requirement
  Word: Type      → JSON: type

表 2（测试需求）
  Word: ID                  → JSON: id
  Word: Name                → JSON: name
  Word: Description/Objective → JSON: description + objective
  Word: Test Data           → JSON: test_data
  Word: Expected Result     → JSON: expected_result
```

### 3.3 Description/Objective 的分割规则

Word 表 2 的第三列合并存储了 `description` 和 `objective`，回写时按以下规则拆分：

- 若文本中含 `描述:` / `desc:` / `description:` 前缀 → 之后内容归入 `description`
- 若文本中含 `目标:` / `obj:` / `objective:` 前缀 → 之后内容归入 `objective`
- 若无任何前缀 → 整段文本作为 `description`，`objective` 留空

**建议专家书写格式**：
```
描述: 用户在支付页面输入合法正数金额，且该金额等于订单应付总额
目标: 验证金额格式正确且逻辑合规时，允许提交
```

或简洁写法（无前缀时系统默认全段为 description）：
```
用户在支付页面输入合法正数金额，且该金额等于订单应付总额。验证金额格式正确且逻辑合规时，允许提交。
```

### 3.4 Test Data 的解析规则

- 若每行含 `键:值` 格式 → 解析为字典对象
- 若不含冒号或格式不符 → 保留为纯文本字符串

**建议格式**：
```
订单总额：¥286.00
输入金额：286.00
```
→ 解析为 `{ "订单总额": "¥286.00", "输入金额": "286.00" }`

## 四、专家修改操作规范

### 4.1 可修改范围

| 操作 | 允许 | 说明 |
|------|------|------|
| 修改需求名称 | ✅ | 使标题更准确 |
| 修改需求描述 | ✅ | 修正表述、补充细节、消除歧义 |
| 修改测试步骤 | ✅ | 细化操作流程 |
| 修改预期结果 | ✅ | 明确通过标准 |
| 修改测试数据 | ✅ | 补充具体输入值 |
| 修改需求类型 | ✅ | 重新分类 |
| 删除整行 | ✅ | 删除无效/重复需求，JSON 会同步删除 |
| 新增整行 | ✅ | 补充遗漏需求，需填写完整 4/5 列 |
| 修改 ID | ⚠️ | **不推荐**。改 ID 会导致与原始 JSON 无法匹配，视为新增条目 |
| 直接修改 JSON | ❌ | 所有修改应在 Word 中进行，通过回写机制同步 |

### 4.2 修改原则

1. **保留 ID**：尽量不改 ID。如需调整编号，确保新旧 ID 不重复。
2. **以行为单位**：删除 Word 表格中的某行，对应 JSON 中的该条需求会被删除。
3. **新增需完整**：新增行必须填写所有列，空单元格回写后会存为空字符串。
4. **不要改表格结构**：不要增删列、合并单元格、改列名，否则回写解析会失败。
5. **Word 不展示的字段不动**：`extra_info` 等内容在 Word 中看不到，回写时会自动保留，无需担心被覆盖。

## 五、修改后的保存与回写

### 5.1 回写方式概览

系统提供两种回写方式，将 Word 中的修改同步回 JSON：

| 方式 | 适用场景 | 操作 |
|------|----------|------|
| **手动同步** | 修改完成后一次性回写 | 调用 API 或运行脚本 |
| **自动监听** | 边改边同步 | 启动 watcher，Word 保存后自动回写 |

### 5.2 方式一：手动同步（推荐）

修改完 Word 报告后，执行以下操作：

```bash
# 在 ScenReq 根目录执行
python -m manual_edit_sync.run_api
```

然后调用同步接口：

```bash
curl -X POST http://localhost:8011/api/manual-edit/sync \
  -H "Content-Type: application/json" \
  -d '{"report_path": "D:/.../xxx_功能测试报告.docx"}'
```

或在代码中调用：

```python
from manual_edit_sync.word_json_sync import sync_word_to_json

result = sync_word_to_json("D:/.../xxx_功能测试报告.docx")
print(f"SRS 更新 {result.srs_updated} 条，Tests 更新 {result.tests_updated} 条")
```

**回写逻辑**：
- 读取 Word 的两个表格
- 按 `id` 与原始 JSON 匹配
- 更新可映射字段
- `extra_info` 等 Word 未展示字段保留原值
- Word 中删除的行，JSON 中同步删除
- 保存回原始 JSON 文件

### 5.3 方式二：自动监听

启动监听服务，Word 文件每次保存后自动触发回写：

```bash
# 启动监听
curl -X POST http://localhost:8011/api/manual-edit/watch/start \
  -H "Content-Type: application/json" \
  -d '{
    "report_path": "D:/.../xxx_功能测试报告.docx",
    "poll_interval_seconds": 2.0
  }'

# 停止监听
curl -X POST http://localhost:8011/api/manual-edit/watch/stop \
  -H "Content-Type: application/json" \
  -d '{"report_path": "D:/.../xxx_功能测试报告.docx"}'
```

> 自动监听每 2 秒检测文件修改时间，发现保存后自动回写。

### 5.4 回写结果确认

回写成功后，接口返回：

```json
{
  "ok": true,
  "report_path": "D:/.../xxx_功能测试报告.docx",
  "srs_json_path": "D:/.../xxx_SRS.json",
  "tests_json_path": "D:/.../xxx_TestRequirements.json",
  "srs_count": 5,
  "tests_count": 12,
  "srs_updated": 3,
  "tests_updated": 8,
  "timestamp": 1715073000.0
}
```

| 字段 | 含义 |
|------|------|
| `srs_count` / `tests_count` | 回写后 JSON 中的总条数 |
| `srs_updated` / `tests_updated` | 实际发生修改的条数 |

## 六、完整协作流程（给专家的步骤）

```
Step 1: 系统生成
  ↓ 系统输出：xxx_SRS.json + xxx_TestRequirements.json + xxx_功能测试报告.docx

Step 2: 专家接收 Word 报告
  ↓ 打开 xxx_功能测试报告.docx

Step 3: 专家修改
  ↓ 在 Word 表格中直接编辑：改文字、删行、加行
  ↓ 保存 Word 文件（Ctrl+S）

Step 4: 回写 JSON（二选一）
  ├─ 方式 A：手动调用同步 API
  └─ 方式 B：已启动自动监听，保存即自动回写

Step 5: 验证
  ↓ 检查返回的 updated 数量是否与修改一致
  ↓ 必要时打开 JSON 确认内容

Step 6: 交付
  ↓ 将修改后的 Word + JSON 一并提交
```

## 七、注意事项与常见问题

### 7.1 修改 Word 时不要做的事

| ❌ 禁止操作 | 后果 |
|------------|------|
| 增删列或改列名 | 回写解析失败，报 "format mismatch" |
| 合并单元格 | 表格解析错乱，数据丢失 |
| 改 ID 后不留痕 | 系统视为新条目，原条目被删除 |
| 在表格外写备注 | 不影响 JSON，但可能造成混淆 |

### 7.2 回写失败排查

| 现象 | 原因 | 解决 |
|------|------|------|
| "Report file not found" | 路径错误 | 检查 `report_path` 是否为绝对路径 |
| "format mismatch" | Word 表格结构被破坏 | 恢复原始表格结构（4 列 + 5 列） |
| SRS 条数变少 | 误删行或 ID 改了导致匹配不上 | 检查 ID 是否一致 |
| extra_info 丢失 | 直接改 JSON 而非 Word | 应通过 Word 回写，extra_info 自动保留 |

### 7.3 文件路径规范

- 所有路径使用**绝对路径**
- Word 报告、SRS JSON、Tests JSON 必须在**同一目录**
- 命名规范：`{base}_功能测试报告.docx` → `{base}_SRS.json` + `{base}_TestRequirements.json`
